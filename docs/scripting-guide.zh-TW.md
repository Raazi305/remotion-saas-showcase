# 📋 展示影片腳本規劃指南

繁體中文 | **[English](./scripting-guide.md)**

一份實用的 SaaS 展示影片規劃、腳本撰寫與製作指南 — 萃取自真實的製作經驗。

---

## 1. 腳本結構

每支展示影片都遵循一致的敘事弧線。使用以下框架規劃場景：

| 階段           | 目的               | 時長建議         |
| -------------- | ------------------ | ---------------- |
| **吸引注意力** | 抓住目光，陳述痛點 | 5–10 秒          |
| **核心流程**   | 走過主要使用者旅程 | 30–90 秒         |
| **邊際情境**   | 展示彈性與自助功能 | 10–20 秒（選用） |
| **行動呼籲**   | 總結價值，引導行動 | 5–10 秒          |

### 小技巧

- **從痛點切入**，而不是產品。「在群組裡文字接龍訂餐根本是場災難」比「歡迎使用我們的 App」有效得多。
- **一個場景 = 一個概念。** 如果一個場景要展示兩個功能，就拆開它。
- **給觀眾喘息空間。** 在重大轉場之間加入 1-2 秒的「靜止時間」— 感覺更自然，也有助於配音節奏。

## 2. 章節設計模式

### 模式 A：雙角色對話

設定兩個人物 — **引導者**與**新手**。這能創造自然的教學流程。

```text
引導者：「你看 — 我只要點這裡，訂單就完成了。」
新手：「等等，如果我之後想改怎麼辦？」
引導者：「沒問題，截止前都能按『編輯』修改。」
```

**適用場景：** 教學、新手引導、使用者端展示。

### 模式 B：旁白 + 螢幕錄製

由單一旁白解說，動畫畫面負責表現。

**適用場景：** 管理後台、儀表板、技術功能展示。

### 模式 C：問題 → 解決方案蒙太奇

快速輪播痛點（每個 3–5 秒），然後乾淨地轉場到產品一一解決它們。

**適用場景：** 行銷影片、Landing Page 主視覺。

## 3. 配音製作

### AI 語音生成

快速原型製作時，**Google NotebookLM**、**ElevenLabs**、**OpenAI TTS** 等 AI 工具能生成自然的配音。

#### NotebookLM Prompt 模板

```markdown
[系統提示：生成一段「深度探討」Podcast 風格的雙人對話]

**角色**：兩位主持人 — 活潑、有洞察力、專業但平易近人。
**語言**：繁體中文
**主題**：[你的產品] — [一句話描述]
**目標**：解釋「價值」與「理念」，而非操作步驟。

## 結構指引
### 階段 1：WHY（願景）
- 痛點：[描述問題]
- 核心價值：[這個產品為何存在]

### 階段 2：HOW（方法）
- 關鍵差異化：[2-3 個重點]

### 階段 3：WHAT（功能）
- 以真實情境類比來介紹功能群組

### 階段 4：結論
- 串連所有概念，行動呼籲
```

### 配音 ↔ 動畫同步

| fps | 1 秒  | 一般句子（3–4 秒） | 段落（8–10 秒） |
| --- | ----- | ------------------ | --------------- |
| 30  | 30 幀 | 90–120 幀          | 240–300 幀      |
| 60  | 60 幀 | 180–240 幀         | 480–600 幀      |

**黃金法則：** 永遠先錄製/生成配音，再製作動畫來配合。千萬不要試圖把配音塞進預設好時間的動畫裡。

#### 工作流程

1. 撰寫腳本
2. 生成配音音檔
3. 測量每段音檔時長
4. 為每個 `<Sequence>` 設定對應的 `durationInFrames`
5. 在時間限制內建構動畫

## 4. 場景時間與節奏

### 游標動畫節奏

| 動作         | 建議時長               | 備註                    |
| ------------ | ---------------------- | ----------------------- |
| 移動到目標   | 0.5–0.8 秒（15–24 幀） | 使用彈簧緩動，避免線性  |
| 點擊         | 0.2 秒（6 幀）         | 短暫的縮放 + 透明度脈衝 |
| 輸入一個字元 | 0.05–0.1 秒（2–3 幀）  | 微幅變化以增加真實感    |
| 動作後停頓   | 0.3–0.5 秒（9–15 幀）  | 讓觀眾看到結果          |
| 場景轉場     | 0.5–1.0 秒（15–30 幀） | 淡入淡出、滑動或彈簧    |

### 場景時長建議

| 場景類型     | 時長範圍 |
| ------------ | -------- |
| 簡單表單填寫 | 5–8 秒   |
| 儀表板總覽   | 8–12 秒  |
| 多步驟流程   | 15–30 秒 |
| 完整功能展示 | 30–60 秒 |

## 5. 製作檢查清單

### 前期製作

- [ ] 確定目標觀眾（終端使用者？管理者？投資人？）
- [ ] 撰寫腳本大綱與章節分段
- [ ] 決定配音風格（對話式 / 旁白 / 純音樂）
- [ ] 列出所需的所有畫面與狀態
- [ ] 確定影片總時長預算

### 製作中

- [ ] 獨立建構場景 — 單獨測試每個 `<Composition>`
- [ ] 錄製/生成配音並測量時間
- [ ] 將 `<Sequence>` 時長對齊配音時間戳
- [ ] 游標動畫最後加入（依賴於版面定稿）
- [ ] 加入背景音樂，音量壓低（比配音低 -15 至 -20 dB）

### 後期製作

- [ ] 以 1x 速度完整觀看 — 確認節奏自然
- [ ] 確認所有文字可讀（至少在畫面上停留 2 秒）
- [ ] 檢查明暗背景下的色彩對比度
- [ ] 以目標解析度輸出（YouTube 用 1920×1080，手機用 1080×1920）
- [ ] 如目標為國際觀眾，加入字幕

## 6. 常見陷阱

| 陷阱               | 解法                                               |
| ------------------ | -------------------------------------------------- |
| 動畫太快           | 放慢 30%。開發時覺得剛好的速度，播放時會覺得太趕。 |
| 一支影片塞太多功能 | 選 3–5 個關鍵功能就好，剩下的留給第二集。          |
| 配音與畫面不同步   | 先錄配音，再做動畫。                               |
| 沒有視覺層次       | 使用 `Indicator` 元件來引導目光。                  |
| 忘記手機觀看者     | 以 50% 縮放測試可讀性。                            |
| 背景音樂太大聲     | 保持在配音下方 -15dB 或更低。                      |
